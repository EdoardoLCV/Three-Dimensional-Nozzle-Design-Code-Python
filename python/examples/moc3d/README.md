# 3D_MOC examples

Three sample wall contours that you can run through the 3D MOC solver,
plus the optional ``SL.inp`` streamtube selection file.

## Files

| File             | What it is                                                 |
|------------------|-------------------------------------------------------------|
| `cone10.geo`     | 10° cone, M=4 design (217 axial stations)                  |
| `M4perfect.geo`  | Perfect (Rao-truncated) M=4 nozzle (162 stations)          |
| `m4Rao.geo`      | Minimum-length M=4 Rao nozzle (162 stations)               |
| `SL.inp`         | Optional list of streamline indices to extract as a tube   |

All three are stacked-circle wall contours: each row is
``z  r  xc  yc`` and defines a circle of radius ``r`` centred at
``(xc, yc)`` in the cross-section at axial station ``z``. For these
axisymmetric examples ``xc = yc = 0`` everywhere.

## Run it

```bash
pynozzle-moc3d cone10.geo --z-step 1 -o out
```

That writes the full set of Tecplot files into ``out/``:

* `Initial Wall.plt`   — the input wall, re-emitted for plotting
* `z=0.out`            — the initial reference plane
* `full_mesh.plt`      — every (i, k) MOC grid point with all flow state
* `axialStations.plt`  — same data grouped by axial plane (zone per k)
* `Streamlines.plt`    — same data grouped by streamline (zone per i)
* `Wall.plt`           — body-point rows only
* `outfile.out`        — body-point unit-normal log
* `streamtube.plt`     — only present if `SL.inp` is in the working dir

Flow inputs default to the GUI defaults (p₀=1000 psia, T₀=530 °R,
M₀=1.1, MW=28.96, γ=1.4, θ=ψ=0, nDiv=36, "All Point Spline"); override
any of them on the command line with ``--p0``, ``--t0``, ``--mach0``,
etc.

The full 217-plane cone10 march takes about 40 seconds on a modern
laptop. M4Perfect and M4RAO at 162 stations take ~30 seconds each.

## A note on the numbers

Unlike the bit-for-bit reproduction we get out of the 2D MOC and the
streamline tracer, the 3D MOC is an *iterative* solver — Newton-Raphson
at every body point, fixed-point iteration at every field point,
thin-plate-spline interpolation on each plane — so tiny floating-point
differences between MSVC's libm and Python's accumulate over the
200+-plane march. Flow-field quantities agree with the original sample
to **4-5 significant digits**, which is the practical noise floor for
this kind of solver. See `README.md` at the top level for the full
breakdown.

## A note on the ``z=0.out`` header

The C++ source writes a ``Radius(in)`` column right after ``Z(in)`` in
``z=0.out``, but the sample outputs were generated by an earlier build
that didn't yet include it. The port omits the column to match the
sample. If you want it back, change the header and the row format in
``pynozzle/moc3d/solver.py``.
